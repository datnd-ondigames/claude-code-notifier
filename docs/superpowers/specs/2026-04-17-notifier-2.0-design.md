# Claude Code Notifier 2.0 — Design Spec

**Date:** 2026-04-17
**Target version:** 2.0.0
**Status:** Approved, pending implementation plan

## Goals

Transform the current single-file VS Code extension (file-watch → warning toast) into a full-featured notification dashboard with guided onboarding.

Two drivers:

1. **UX polish** — settings UI, status bar, history panel, action buttons, focus-aware suppression, per-event styling, built-in sound.
2. **Onboarding** — first-run wizard that installs Claude Code hooks automatically, no manual JSON editing.

Out of scope for 2.0: Windows support, test suite, marketplace publishing automation, telemetry.

## Non-goals

- Replacing `terminal-notifier` or any OS notifier beyond the extension's own delivery.
- Supporting remote / multi-machine setups. Dashboard binds `127.0.0.1` only.
- Persisting notification history across restarts beyond the log file itself. In-memory ring buffer is authoritative for the UI.

## Architecture overview

```
Claude hook ──writes JSON line──▶ ~/.claude/notifier.log
                                      │
                              log-tailer (fs.watch + byte offset tail)
                                      │
                                      ▼
                              history-store (ring buf + emitter)
                       ┌──────────────┼──────────────┐
                       ▼              ▼              ▼
                  notifier.js    server (SSE)    webview panel
                   (VS Code       (external       (in-editor
                    toast +       browser          tab, mirrors
                    sound +       dashboard)       dashboard)
                    actions)
                       │
                action click ──▶ vscode.commands / OS focus
```

One instance of each module runs per VS Code window. Multiple windows = multiple tailers on same log + multiple HTTP servers on separate ports.

### Module layout

```
claude-code-notifier/
├── extension.js              # thin bootstrap → src/activation.js
├── package.json
├── media/
│   ├── dashboard.html        # served by server AND loaded by webview
│   ├── dashboard.js          # SSE client, render, focus POST
│   ├── dashboard.css
│   ├── webview.html          # webview shell, imports dashboard assets
│   └── sounds/*.wav
└── src/
    ├── activation.js         # VS Code entry, wires modules, registers cmds
    ├── config.js             # reads settings, typed defaults, change events
    ├── log-tailer.js         # tail notifier.log, emit parsed events
    ├── history-store.js      # ring buffer (maxEntries), event bus, unread set
    ├── notifier.js           # toast dispatch, action buttons, sound trigger
    ├── sound.js              # play .wav via OS command (afplay / paplay)
    ├── server.js             # http server: /, /static, /events SSE, /focus, /dismiss, /history
    ├── webview.js            # internal WebviewPanel mirror
    ├── wizard.js             # first-run flow
    ├── hook-writer.js        # read/merge/write settings.local.json + emit helper
    └── status-bar.js         # unread badge + URL + click handler
```

Every module <200 lines, pure JS, no bundler, no TypeScript.

## Protocol

### Log format

`~/.claude/notifier.log` (override: `claudeNotifier.logPath`). NDJSON, append-only.

**Event schema:**

```json
{
  "id": "01HXK3ABCDEF1234567890",
  "ts": 1713340800123,
  "type": "permission_prompt",
  "msg": "Claude needs your permission",
  "cwd": "/home/felix/Git-projects/foo",
  "session": "abc123",
  "pid": 48210
}
```

| Field   | Required | Meaning |
|---------|----------|---------|
| `id`    | yes      | Unique event id (timestamp + random, ~22 chars). Dedup key. |
| `ts`    | yes      | Epoch ms. |
| `type`  | yes      | One of: `permission_prompt`, `elicitation_dialog`, `idle_prompt`, `stop`, `subagent_stop`, `custom`. |
| `msg`   | yes      | Human text. |
| `cwd`   | no       | Workspace path for multi-window routing. |
| `session` | no     | Claude session id. |
| `pid`   | no       | Claude process pid. |

### Emit helper

`~/.claude/notifier-emit.js`, installed by wizard, node-stdlib only:

```js
#!/usr/bin/env node
const fs = require('fs');
const path = require('path');
const os = require('os');

const [,, type, ...msgParts] = process.argv;
const msg = msgParts.join(' ');
const logPath = process.env.CLAUDE_NOTIFIER_LOG
  || path.join(os.homedir(), '.claude', 'notifier.log');

const newId = () => {
  const t = Date.now().toString(36);
  const r = Math.random().toString(36).slice(2, 12);
  return (t + r).padEnd(22, '0').slice(0, 22).toUpperCase();
};

const entry = {
  id: newId(),
  ts: Date.now(),
  type,
  msg,
  cwd: process.env.PWD || process.cwd(),
  session: process.env.CLAUDE_SESSION_ID || null,
  pid: process.pid
};

try {
  fs.mkdirSync(path.dirname(logPath), { recursive: true });
  fs.appendFileSync(logPath, JSON.stringify(entry) + '\n');
} catch (e) {
  process.stderr.write(`notifier-emit: ${e.message}\n`);
  process.exit(1);
}
```

Helper is versioned via inline `_version` constant; extension rewrites on activation if shipped version is newer.

### Tailing

`log-tailer` tracks byte offset across `fs.watch` events. On change: read offset → EOF, split on `\n`, parse each line, skip parse errors (logged to output channel), skip IDs already in history-store. Initial load: read last N bytes (capped to last 1000 lines) to backfill history.

### Log rotation

On activation, if log > 5 MB: read last 1000 lines into memory, truncate file, write them back. Bounded disk use without data loss for recent events.

### Dedupe

History-store rejects events where `id` already present OR where `type + msg + cwd` matches an entry within the last 2 seconds.

## First-run wizard

**Trigger:** on activation, check `context.globalState.get('claudeNotifier.wizardCompleted')`. If absent or false → show wizard. Command `Claude Notifier: Run Setup Wizard` reopens anytime.

**Steps** (VS Code `showQuickPick` / `showInputBox` sequence):

1. **Scope** — install hooks where?
   - Global (`~/.claude/settings.local.json`) — all projects
   - Workspace (`.claude/settings.local.json` in current folder)
   - Both

2. **Preset** — pick event set:
   - **Minimal** — `permission_prompt` + `elicitation_dialog`
   - **Standard** — Minimal + `idle_prompt`
   - **All** — + `stop` + `subagent_stop`
   - **Custom** → opens step 2b

2b. **Custom events** — multi-select checkbox list of all five types, pre-checked per last-used preset.

3. **Sound** — play sound on notification? Yes (default) / No.

4. **Confirm** — show preview of hooks JSON and target path(s). Buttons: `Install` / `Cancel` / `Copy to clipboard`.

**On Install:**

1. `hook-writer` reads or creates target `settings.local.json`.
2. Merges `hooks.Notification[]` (and `hooks.Stop[]`, `hooks.SubagentStop[]` if selected) — skips entries already flagged managed (see below), adds missing.
3. Writes/updates `~/.claude/notifier-emit.js`.
4. Persists log path setting if overridden.
5. Sets `wizardCompleted = true`.
6. Shows toast: "Hooks installed. Restart Claude Code for changes to apply."

**Uninstall path:** command `Claude Notifier: Remove Hooks` — walks known settings paths, removes every hook entry whose command string contains the sentinel comment `# claude-code-notifier` (see below). Optionally prompts to delete helper + log.

### Managed-entry sentinel

Since Claude's hook schema may not preserve arbitrary JSON keys, the managed-entry marker lives inside the command string itself as a trailing shell comment:

```json
{
  "type": "command",
  "command": "node ~/.claude/notifier-emit.js permission_prompt 'Claude needs your permission' # claude-code-notifier"
}
```

Uninstall and idempotent re-install match on the `# claude-code-notifier` substring in the command. This is robust across Claude schema changes and survives JSON serialization round-trips.

## Webview panel + external dashboard

Shared frontend: `media/dashboard.html`, `dashboard.js`, `dashboard.css`. Same HTML served from both transports; client-side mode detection picks transport.

### Internal webview (`src/webview.js`)

- Command: `Claude Notifier: Open Panel` → `vscode.window.createWebviewPanel('claudeNotifier', 'Claude Notifier', ViewColumn.Beside, { enableScripts: true, retainContextWhenHidden: true })`.
- HTML bootstrap injects `<script>window.__MODE = 'webview'</script>`.
- Extension pushes events via `panel.webview.postMessage({ type: 'event', payload })` when history-store emits.
- Row click → `postMessage({ type: 'focus', id })` back to extension → `workbench.action.focusActiveEditorGroup` + terminal reveal if present.

### External browser dashboard (`src/server.js`)

- `http.createServer` on free port starting 37100, incrementing per window.
- Routes:
  - `GET /` — serves `dashboard.html` with `<script>window.__MODE = 'browser'; window.__PORT = <port>;</script>`.
  - `GET /static/*` — serves `dashboard.js`, `dashboard.css`, sounds.
  - `GET /history` — JSON array of current ring buffer, newest first.
  - `GET /events` — SSE stream; pushes new events as `data: {json}\n\n`.
  - `POST /focus` — body `{ id }` — extension focuses VS Code window via `child_process.execFile` platform-specific (`wmctrl -a <title>` on Linux, `osascript -e ...` on macOS) + `workbench.action.focusActiveEditorGroup`. Arguments passed as arg array, never shell-interpolated.
  - `POST /dismiss` — body `{ id }` — marks event read in history-store, broadcasts to all SSE clients.
- Binds `127.0.0.1` only. No CORS, no auth (localhost trust model).

### UI layout (shared)

- **Header:** workspace label, filter chips per event type, search box, "mark all read", "clear", sound-on-arrival toggle.
- **Body:** virtualized list, newest top. Each row: type icon, HH:MM:SS time, message, workspace path, unread dot.
- **Footer:** SSE connection status indicator, port, event count.
- **Realtime:** new events slide in at top. Optional arrival pling (separate from notifier.js sound).

### Transport abstraction in `dashboard.js`

```js
const mode = window.__MODE;
const transport = mode === 'webview'
  ? new VsCodeTransport(acquireVsCodeApi())
  : new SseTransport('/events', '/focus', '/dismiss');
```

Transport interface: `onEvent(cb)`, `sendFocus(id)`, `sendDismiss(id)`, `fetchHistory()`.

## Settings schema

`package.json` → `contributes.configuration`:

- `claudeNotifier.enabled` (boolean, default `true`)
- `claudeNotifier.logPath` (string, default `""` → `~/.claude/notifier.log`)
- `claudeNotifier.suppressWhenFocused` (boolean, default `false`) — skip toast + sound if VS Code window focused; event still lands in history/dashboard.
- `claudeNotifier.showAllWorkspaces` (boolean, default `false`) — if false, toasts fire only for events whose `cwd` matches this window's workspace folder.

**Per event type** (`permission_prompt`, `elicitation_dialog`, `idle_prompt`, `stop`, `subagent_stop`):

- `claudeNotifier.events.<type>.enabled` (boolean)
- `claudeNotifier.events.<type>.severity` (`info` | `warning` | `error`)
- `claudeNotifier.events.<type>.sound` (boolean)

**Defaults:**

| Type                | enabled | severity | sound |
|---------------------|---------|----------|-------|
| permission_prompt   | true    | warning  | true  |
| elicitation_dialog  | true    | warning  | true  |
| idle_prompt         | false   | info     | false |
| stop                | false   | info     | false |
| subagent_stop       | false   | info     | false |

**Other:**

- `claudeNotifier.history.maxEntries` (number, default `500`, min 50, max 10000)
- `claudeNotifier.statusBar.enabled` (boolean, default `true`)
- `claudeNotifier.statusBar.clickAction` (`dashboard` | `panel` | `quickPick`, default `dashboard`)
- `claudeNotifier.actions.showFocusWindow` (boolean, default `true`)
- `claudeNotifier.actions.showOpenTerminal` (boolean, default `true`)
- `claudeNotifier.actions.showSnooze` (boolean, default `false`)
- `claudeNotifier.snoozeMinutes` (number, default `5`, min 1, max 60)
- `claudeNotifier.server.enabled` (boolean, default `true`)
- `claudeNotifier.server.port` (number, default `0` → auto-pick starting 37100)

**Change handling:** `config.js` subscribes to `vscode.workspace.onDidChangeConfiguration`. On change: re-reads, emits `configChanged`. Modules react — server restarts if port/enabled changed, status bar rebuilds, notifier regenerates action list, etc.

## Commands

`contributes.commands`:

- `claudeNotifier.notify` — test notification (legacy id, kept).
- `claudeNotifier.openPanel` — internal webview.
- `claudeNotifier.openDashboard` — external URL via `vscode.env.openExternal`.
- `claudeNotifier.runWizard`
- `claudeNotifier.installHooks` — re-run install without full wizard.
- `claudeNotifier.removeHooks`
- `claudeNotifier.clearHistory`
- `claudeNotifier.showLog` — reveal log file in editor.

## Status bar

Single left-aligned item, priority 100.

**States:**

| State | Text | Icon |
|-------|------|------|
| Idle, no unread | `Claude :37100` | `$(bell)` dim |
| Unread N | `Claude (3) :37100` | `$(bell-dot)` highlighted |
| Server error | `Claude ⚠` | `$(bell-slash)` |
| Snoozed | `Snoozed 4m` | `$(bell-slash)` |
| Disabled | hidden | — |

**Tooltip:** lists last 3 unread messages + hint text.

**Click action:** configurable via `statusBar.clickAction`:

- `dashboard` — open external URL (default)
- `panel` — open internal webview
- `quickPick` — menu: Open Dashboard / Open Panel / Run Wizard / Clear History / Open Settings

## Notifier pipeline

`notifier.js` handles each event from history-store:

```
event ──▶ enabled-for-type check
     ──▶ workspace match check (unless showAllWorkspaces)
     ──▶ snooze check (snoozed → history-only)
     ──▶ focus check (suppressWhenFocused && focused → history-only)
     ──▶ severity → vscode.window.show{Info,Warning,Error}Message
     ──▶ action buttons built from settings
     ──▶ sound.play(type) if sound=true
```

### Action buttons

VS Code shows ~3 buttons. Priority order when all enabled:

1. **Focus Window** — `workbench.action.focusActiveEditorGroup` plus OS-level front (Linux `wmctrl`, macOS `osascript`). No-op if already focused. All shell arguments passed via `execFile` arg array, never interpolated into a shell string.
2. **Open Terminal** — `workbench.action.terminal.focus`, creates terminal if none.
3. **Open Dashboard** — `vscode.env.openExternal(Uri.parse('http://127.0.0.1:<port>'))`.
4. **Snooze Nm** — sets `snoozedUntil = now + snoozeMinutes*60_000`. While snoozed: toasts + sounds suppressed, dashboard/webview/history unaffected. Status bar shows snooze state.

Dismiss is implicit via toast X close.

### Focus detection

Initial state from `vscode.window.state.focused`. Kept fresh via `vscode.window.onDidChangeWindowState`.

## Sound

`sound.js` plays bundled `.wav` files via OS commands, dispatched with `child_process.execFile` (fixed argv, no shell).

- Platform detection:
  - macOS: `afplay <path>`
  - Linux: `paplay <path>` (fallback chain `aplay`, `play`) — first available on `PATH` wins, resolved once on module load.
- Bundled `.wav` files per event type in `media/sounds/`: `permission.wav`, `question.wav`, `idle.wav`, `default.wav`.
- Mapping: event type → file (e.g. `permission_prompt` → `permission.wav`). Unknown types → `default.wav`.
- `play(type)`: spawns player with wav path, fire-and-forget. Errors logged to output channel once per missing player; never throws.
- The webview dashboard has its own independent arrival pling via HTML5 `<audio>`, unrelated to this module.

No hidden webview. VS Code extension host invokes the OS audio player directly, so nothing appears in the editor UI.

## Security

- HTTP server binds `127.0.0.1` only; never exposed beyond loopback.
- No shell interpolation anywhere. All OS-focus calls use `child_process.execFile(cmd, [args...])` with fixed argv arrays — no attacker-influenced string goes through a shell.
- SSE and POST endpoints accept only JSON bodies with a known schema; unknown fields dropped.
- Webview `enableScripts: true` but `localResourceRoots` limited to `media/` folder.
- Emit helper is node-only, no shell expansion of message arguments since args come in via `process.argv` array from hook command.

## Error handling

- Log file unreadable → status bar switches to error state, output channel explains, extension stays active.
- Hook install fails (permission, parse error) → wizard shows error with path + reason, offers "Copy JSON to clipboard" so user can edit manually.
- Server port bind fails → tries next port up to 37150, then disables server + surfaces error; webview still works.
- Malformed log line → skipped, logged to output channel, next line processed.
- Helper script missing or stale → rewritten on next activation.

Extension never crashes the window. All errors flow to VS Code output channel `Claude Notifier`.

## Migration from 1.x

Current users have:

- `/tmp/claude-notify` file-watch setup
- Hook commands writing plain text to that file

On 2.0 activation:

1. If `/tmp/claude-notify` exists, detect legacy setup — show one-time info toast: "Claude Notifier 2.0 uses a new log format. Run Setup Wizard to migrate." Offer button → opens wizard.
2. Wizard detects and offers to remove legacy hook entries (match on literal `/tmp/claude-notify` command substring).
3. Legacy file-watch path is removed from code. No attempt to dual-support.

README updated to reflect new setup flow. Old manual-config section moved to "Legacy 1.x" appendix.

## Testing strategy (informal, 2.0 scope)

Automated tests deferred to post-2.0. For 2.0, manual verification per module:

- Wizard: fresh install, reinstall, uninstall, custom selection, both scopes.
- Log tailer: rapid writes, partial lines, log rotation, parse errors.
- Server: port conflict, restart on config change, SSE reconnect.
- Webview + dashboard: event arrival, filter, search, focus click, dismiss.
- Status bar: unread counting, snooze state, error state.
- Notifier: each event type, severity, action buttons, focus suppression, snooze.
- Sound: each sound file, fallback when webview fails.
- Migration: user on 1.x upgrades cleanly.
